THE ORIGIN

Why we are woring on this project.

JavaScript is good at rapid developing of web UI behavior and developing of high performance server-side application, thanks to its flexible function-oriented programming style and non-block single thread model with task queue. But when talking about object-oriented programming, most developers are not satisfied with the constructor/prototype object model of the language.

We suffered a lot from the limited ability of the language to intuitively/naturally describe some important object-oriented concepts and patterns when we were porting a desktop version direct UI library in C++ to a web version in JavaScript recently. These concepts and patterns had to be implemented via various work-around due to the limited describing ability of the constructor/prototype model, which nearly drived us mad.

Constructor/prototype model is lack of many important object-oriented concepts, such as private properties and methods, an independent parent part for individual objects (that is, each object of the same class that is derived from a specific parent class cannot have its own parent part, which may be constructed with different initializing arguments, since they share the same parent object called the prototype, which is constructed fairly early), accessing parent class methods which are overridden by a derived class method, multi-inheritance*, static property and method scopes (which means a parent class static member cannot be accessed through a derived class) and so on.

Efforts were made to address those issues by developers, such as simulating private member behaviors using closure scopes which really works and is widely used, constructing objects by directly calling constructors instead of using a new operator which brings multi-inheritance ability to the language and makes parent parts of individual objects independent but makes overridden methods no longer accessible as a side effect, and so on. Unfortunately, no one has an idea so far about how to make JavaScript completely an intuitive object-oriented language and no one knows a way so far to address all the issues mentioned above.

We are working hard here trying to make JavaScript fully intuitively objected-oriented by going on a brand-new way.

Inheritance

				
The most complicated kid in the object-oriented world and the most concise task in class.js. 

			

			

				

					
Inheritance makes life tough, because it brings a lot of complexity to the object-oriented world. Those include instance method polymorphism and instance property stability (the method with the expected name in the most derived class will be called whenever a method call is made with a name but property accessing with a name in a class will always access the property declared in that class or its base classes no matter whether there is a derived class that declares a property with the same name), constructor chaining (derived classes are responsible for providing arguments to parent classes when constructing instances), parent part accessing (which makes overridden methods and properties shadowed with same-name properties in derived classes accessible), static scopes (which means a static member declared in a parent class can be accessed through its derived classes), multi-inheritance and so on. 


					
But here, class.js makes all those tasks easy. Everything here is just right, natural and intuitive as you expect!

Static scopes

					
A static member declared in a parent class can be accessed through its derived classes. That is also a widely used part of common object-oriented sense. 

					
Class.js goes one step further. All the names in a class namespace scope, including static members, other classes and other namespaces, are accessible through a derived class of this class. Note that this accessibility does not mean a simple copy of all the names in a class to all its derived classes. Modifies through derived classes are accessible through parent classes, too. And vice versa.
